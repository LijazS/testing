context links 3 things ( cluster + user + namespace )

when u do kubectl get pods:
authN and authZ happens

AUTH
- Who are you
- establishing connection to api server and verifying current users identity

  -> kubectl reads .kube config file
  -> from current context , it gets cluster(API server endpoint) , user and ns
  -> establish connection with API server using ca.crt
  -> kubectl sends credintials (depends on sertup)
	( admin.crt and key for local setup, bearer token with kubeadm, tokens with cloud)
  -> API server verifies identity, checks CN,O


authN done
now authZ

- What are you allowed to do.
- After auth api server knows who is trying to commuincate, user and group
- now it checks if the user has access to whatever command they used.

  ->  Scans roles,rolebindings,clluterrole and cluster bindingsa


OIDC

- gets token from an Idp, digitally signed by the idp
- now kubectl verifies user with bearer jwt instead of a crt


Role 
-ns scoped, defines what permissions a user or group needs.

ClusterRole
-cluster scoped,

roleBinding
- ns scoped
- Can be done to assign to users to the defined roles
- both role and cluster role can be binded to this.
- cluster role will also get attached to a ns

ClusterBinding
used to bind users to clusterRole. to get cluster levevl access.
